# Real-Time Attendance Sync Plan

## Problem
Multiple users marking attendance simultaneously leads to duplicate entries because:
- App relies heavily on local database
- No real-time sync with remote server
- Users cannot see updates from other users
- Server rejects duplicates but users have no visibility

## Current Architecture Analysis

### Attendance Marking Flow (from `attendance_screen.dart`)
1. User searches/selects contact
2. `ContactResultCard` calls `onAttendanceMarked` callback (line 487)
3. Callback triggers `_loadMarkedContacts(refreshSearch: false)` (line 487)
4. Actual attendance recording happens in `ContactResultCard` (need to verify implementation)

### Key Files to Examine
- `lib/features/attendance/presentation/widgets/contact_result_card.dart` - attendance marking UI logic
- `lib/core/sync/sync_manager.dart` - existing sync infrastructure
- `lib/core/sync/sync_manager_provider.dart` - sync notifier/state
- `lib/features/contacts/data/repositories/contact_repository_impl.dart` - data layer
- `lib/core/database/database.dart` - local database (Drift)
- Backend API endpoints for attendance

## Proposed Solution

### Phase 1: Immediate Optimistic UI + Server Sync (Short-term)
1. **Optimistic UI Update with Conflict Detection**
   - When user marks attendance, optimistically add to local marked list
   - Immediately attempt server sync
   - If server returns conflict (duplicate), show feedback and refresh from server
   - Rollback optimistic update on failure

2. **Real-time Attendance Refresh**
   - Implement periodic sync while on AttendanceScreen (every 5-10 seconds)
   - Poll for new attendances from other users
   - Update marked contacts list and UI accordingly
   - Use existing `SmartSyncNotifier` active mode (already set on line 45)

3. **Live Presence Indicator**
   - Show indicator when another user has marked attendance recently
   - Optionally show count of new attendances since last sync

### Phase 2: Real-time Push Updates (Medium-term)
1. **WebSocket/Stream Subscription**
   - Establish WebSocket connection to server for attendance events
   - Subscribe to attendance updates for current service/date
   - Push updates to all clients immediately when new attendance recorded
   - More efficient than polling

2. **Server-Side Event Broadcasting**
   - Backend needs to broadcast attendance marking events
   - Include: contact ID, service type, date, user who marked
   - All connected clients receive and update local state

### Phase 3: Conflict Prevention UI (Enhancement)
1. **Pre-check Before Marking**
   - Before recording attendance, do quick server check
   - Show "About to mark" warning if someone else just marked same person
   - Provide option to refresh and see current state

2. **Better User Feedback**
   - Show sync status indicator (pending, synced, conflict)
   - Visual distinction between locally marked vs server-confirmed
   - Undo/Cancel option for optimistic marks before sync completes

## Technical Implementation Steps (Phase 1)

### Step 1: Modify Attendance Recording Flow
- Update ContactResultCard to:
  - Call optimistic UI update immediately
  - Attempt server sync
  - Handle success/error responses
  - Trigger refresh on conflict

### Step 2: Add Periodic Sync to AttendanceScreen
- In `initState`, start a timer that calls `_loadMarkedContacts` every 5-10 seconds
- Ensure `_loadMarkedContacts` fetches from server (not just local DB)
- Cancel timer in `dispose`

### Step 3: Ensure `_loadMarkedContacts` Fetches Server Data
- Currently uses `database.getAttendancesByDateRange` (local only)
- Need to call backend API to get latest attendances
- Merge with local or replace depending on architecture

### Step 4: Marked Contact State Management
- markedContactIdsProvider should reflect server-confirmed state
- Optimistic updates: add to provider immediately
- On sync failure: remove and show error

### Step 5: UI Improvements
- Add sync status indicator
- Show "Syncing..." spinner
- Show conflict error messages

2. **Sync Frequency**: For Phase 1 polling, what interval would you prefer? (5s, 10s, 30s)? Balance between freshness and server load.

3. **Conflict Handling Priority**: Should we prioritize preventing duplicates (pre-check before marking) or detecting them after (rollback on conflict)?
